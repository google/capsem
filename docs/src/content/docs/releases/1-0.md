**1.0.1778378133 | 2026-05-10**

This release completes the Policy V2 sprint: named policy rules now cover MCP,
HTTP, DNS, model request/response, and model tool-call/tool-response runtime
paths. Policy Hook Spec0 ships as a contract, client, and audit foundation; it
is not exposed as configured external hook dispatch in the release UI/runtime.

## Highlights

### Policy V2

Rules live under `policy.<type>.<rule_name>` and use typed callbacks, a strict
condition subset, `allow`/`ask`/`block`/`rewrite` decisions, priorities,
rewrite targets, and audit reasons. User and corporate policy files merge with
corp precedence.

### MITM Enforcement

The framed MCP, HTTP, DNS, and model MITM paths now enforce configured policy
before unsafe dispatch or guest delivery. Denied and rewritten paths redact
secret-bearing previews before `session.db` writes.

### Policy Hooks

Policy Hook Spec0 is exported as OpenAPI 3.1 at
`config/policy-hook-openapi.json` and from `GET /policy-hook/spec`. The
hardened hook client enforces strict endpoint config, HTTPS plus bearer auth for
non-local endpoints, bounded response bodies, fail-closed fallback decisions,
and `policy_hook_events` audit rows with error/fallback state. Configured
external hook dispatch is a follow-up integration path, not a shipped user
control in this release.

### Verification

Release prep added deterministic VM E2E coverage for model response
block/rewrite and provider-emitted tool-call block/rewrite using a local
OpenAI-shaped upstream fixture, plus Criterion microbenchmarks for HTTP, DNS,
model, hook policy matching, and hook response decoding.
