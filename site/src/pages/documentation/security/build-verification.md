Every Capsem release ships with cryptographic proof of what was built, how it was built, and what dependencies it contains. This page explains how to verify a release.

## Release artifacts

Each GitHub release includes:

| Artifact | Purpose |
|---|---|
| `Capsem_X.Y.Z_aarch64.dmg` | Signed and notarized macOS app bundle |
| `Capsem.app.tar.gz` | Compressed app bundle for Tauri auto-updater |
| `Capsem.app.tar.gz.sig` | Minisign signature for the updater bundle |
| `rootfs.squashfs` | VM root filesystem (downloaded on first launch) |
| `manifest.json` | Version manifest with BLAKE3 hashes for all assets |
| `manifest.json.minisig` | Minisign signature for the manifest |
| `capsem-sbom.spdx.json` | Software Bill of Materials (SPDX 2.3 format) |

## Build attestation

Capsem uses [GitHub Attestations](https://docs.github.com/en/actions/security-for-github-actions/using-artifact-attestations) to cryptographically bind each DMG to the CI workflow that produced it. Two attestations are generated per release:

- **Build provenance** (SLSA) -- proves which commit, workflow, and runner produced the artifact
- **SBOM attestation** -- links the SPDX dependency list to the exact artifact

### Verifying a release

Download the DMG from the [releases page](https://github.com/google/capsem/releases), then verify both attestations:

```bash
# Verify build provenance
gh attestation verify Capsem_*.dmg --repo google/capsem

# Verify SBOM attestation
gh attestation verify Capsem_*.dmg --repo google/capsem \
  --predicate-type https://spdx.dev/Document
```

Both commands check the Sigstore signature chain against the GitHub-hosted transparency log. No keys to manage -- verification is based on the CI workflow identity.

## Manifest signatures

The `manifest.json` file contains BLAKE3 hashes for every VM asset (kernel, initrd, rootfs). It is signed with [Minisign](https://jedisct1.github.io/minisign/):

```bash
# Install minisign
brew install minisign

# Verify manifest signature
minisign -Vm manifest.json -p manifest-sign.pub
```

The public key is published in the repository at `config/manifest-sign.pub` and embedded in the app binary.

## SBOM

The SBOM (`capsem-sbom.spdx.json`) is generated by `cargo sbom` in SPDX 2.3 JSON format. It lists every Rust crate compiled into the release binary, including version and license.

You can inspect it with any SPDX-compatible tool:

```bash
# View with jq
jq '.packages[] | {name, versionInfo, licenseDeclared}' capsem-sbom.spdx.json

# Or use an SBOM viewer
npx @anchore/sbom-viewer capsem-sbom.spdx.json
```

## Code signing and notarization

The macOS app bundle is:

1. **Code-signed** with a Developer ID Application certificate (Apple-issued)
2. **Notarized** by Apple's notary service (malware scan + ticket stapling)
3. **Packaged** as a DMG with Gatekeeper clearance

macOS verifies the signature and notarization ticket automatically on first launch. You can also verify manually:

```bash
# Check code signature
codesign -dvv /Applications/Capsem.app

# Check notarization
spctl --assess --type execute /Applications/Capsem.app
```

## CI pipeline

The full release pipeline runs in GitHub Actions on `macos-14` runners:

1. **Preflight** -- validates Apple certificate, Tauri signing key, and notarization credentials
2. **Build assets** -- compiles Linux kernel and rootfs on `ubuntu-24.04-arm` (native arm64)
3. **Test** -- dependency audit, unit tests, cross-compile check, frontend type-check and build
4. **Build app** -- compiles Tauri binary with `cargo-auditable`, code-signs, notarizes, generates SBOM, creates attestations

The `cargo-auditable` tool embeds dependency metadata directly in the binary, enabling post-hoc vulnerability scanning with `cargo audit bin`.

## Reproducible builds

The VM assets (kernel, rootfs) are built in Docker containers from pinned base images, producing deterministic output. The kernel config is checked into the repository at `images/defconfig.arm64`. Anyone can rebuild the assets:

```bash
just build-assets
```

The rootfs is built from `images/Dockerfile.rootfs` with pinned package versions. The kernel is compiled from a specific LTS release fetched from kernel.org.
